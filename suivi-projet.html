Suivi de projet

Cette partie montre comment mon travail est organisé dans un cadre professionnel : User Stories, tâches techniques, estimations, sprint, daily, suivi des points restants et Pull Requests.

Découper une User Story en tâches techniques

Comprendre une User Story Créer des tâches techniques Estimer la difficulté

La trace 4 montre l'organisation du travail dans Azure DevOps. Une User Story correspond à une tâche fonctionnelle. Pour avancer concrètement, je dois ensuite la découper en tâches techniques plus précises.

Ce fonctionnement m'a permis de développer le savoir-faire traduire un besoin fonctionnel en tâches techniques. Je dois comprendre ce qui est demandé, identifier les étapes de développement, puis estimer la difficulté avec l'équipe.

Les estimations sont discutées avec les collègues. Chacun donne une note de 0 à 10, puis l'équipe échange pour identifier les difficultés possibles. Cette méthode m'aide à mieux prévoir mon travail, même si l'organisation et l'anticipation restent des axes de progression pour moi.

Améliorer son code grâce aux Pull Requests

Soumettre son code Corriger après revue Accepter les retours

La trace 5 présente une Pull Request. Lorsqu'une fonctionnalité est terminée, le code peut être relu par un collègue. Cette relecture permet de repérer des erreurs, de proposer une meilleure solution ou d'améliorer la lisibilité du code.

Ce savoir-faire est important car le développement en entreprise ne consiste pas seulement à faire fonctionner son code. Il faut aussi produire un code compréhensible, maintenable et conforme aux règles de l'équipe.

Au départ, recevoir des retours peut être frustrant, car cela donne l'impression que le travail n'est pas terminé. Avec l'expérience, j'ai compris que la revue de code est surtout un moyen de progresser et d'éviter que des erreurs arrivent en production.

Suivre l'avancement d'un sprint

Suivre les points restants Identifier les blocages Rendre compte de son activité

La trace 6 montre le suivi d'avancement du sprint. Les tâches sont estimées en points, ce qui permet de suivre la quantité de travail restante plutôt que seulement le nombre de tâches terminées.

Ce fonctionnement m'aide à mieux visualiser l'état du projet. Tous les matins, la daily permet de dire ce qui a été fait la veille, ce qui est prévu aujourd'hui et les éventuels blocages.

Cette méthode m'a fait progresser sur le savoir-faire rendre compte de mon activité. C'est encore un point que je dois améliorer, car j'ai parfois du mal à valoriser ce que j'ai fait et je vois plus facilement les difficultés que les réussites.

Savoir-faire général : travailler dans une organisation projet

L'alternance chez 3maGroup m'a permis de découvrir une organisation professionnelle avec Azure DevOps, les User Stories, les sprints, les estimations, les dailies et les Pull Requests. Ces méthodes donnent un cadre au développement et permettent de mieux suivre l'avancement du travail.

Avant l'alternance, j'avais des difficultés à organiser mon travail, à planifier et à prévoir les étapes d'un projet. Aujourd'hui, je comprends mieux comment découper une demande en tâches et comment rendre compte de mon avancement. Mon niveau est correct mais encore perfectible : je progresse, mais je dois encore améliorer mon anticipation et ma proactivité.

Savoir-faire général : collaborer autour du code

Les Pull Requests m'ont appris que le code est un travail collectif. Les retours de collègues me permettent d'améliorer mes solutions et de respecter les conventions internes. Ce savoir-faire a été appris principalement en entreprise, car il dépend fortement des méthodes de l'équipe.

Je suis plus à l'aise qu'au début pour demander de l'aide, expliquer un problème et accepter des corrections. Je dois encore progresser dans ma capacité à analyser mon code avant de le soumettre.
